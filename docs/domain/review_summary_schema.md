# Review Summary Schema for VAT Pre-Submission Review Prototype

## 1. Purpose

This document defines the schema for the pre-submission review summary generated by the prototype after record checking and manual review activity.

The review summary is intended to provide a structured overview of:

- what dataset was reviewed
- how many records were processed
- what issues were detected
- how those issues were classified
- what review actions were taken
- what remains unresolved or escalated before VAT preparation continues

The review summary is a reporting and traceability artefact. It is not a statement of final VAT liability, legal compliance certification, or automatic approval for submission.

## 2. Role in the Workflow

The review summary is the final output stage in the pre-submission review workflow.

It should be generated after:

1. spreadsheet import
2. data normalisation
3. rule execution
4. issue generation
5. issue classification by status and risk
6. user review and decision logging
7. issue review-state updates

The summary should consolidate system-detected findings and human review outcomes into one clear output suitable for:

- on-screen review
- export
- dissertation demonstration
- audit-style explanation of what the prototype did

## 3. Design Principles

The review summary schema should follow these principles:

- **Concise but informative**: it should give a clear high-level view without reproducing every issue in full.
- **Traceable**: summary counts should be derived from underlying records, issues, and decisions.
- **Explainable**: the summary should make it clear what was reviewed and what remains unresolved.
- **Human-in-the-loop aware**: it should distinguish between system detections and user-reviewed outcomes.
- **Non-overclaiming**: it should not state that the dataset is fully compliant or legally safe to submit.

## 4. Core Entity Definition

A review summary represents one completed or in-progress pre-submission review session for one dataset or review batch.

The summary should typically aggregate:

- dataset metadata
- record-level processing counts
- issue counts
- issue status distribution
- issue risk distribution
- decision outcome distribution
- unresolved and escalated counts
- timestamps and completion indicators

In a simple implementation, one imported dataset produces one summary.

In a more advanced implementation, multiple summary snapshots may exist over time as the user continues reviewing issues.

For an undergraduate prototype, one current summary per review batch is sufficient.

## 5. Required Fields

Each review summary should include the following required fields.

| Field | Type | Required | Description |
|---|---|---|---|
| `summary_id` | string | Yes | Unique identifier for the review summary |
| `dataset_id` | string | Yes | Identifier of the reviewed dataset or batch |
| `generated_at` | datetime string | Yes | Timestamp when the summary was generated |
| `total_records` | integer | Yes | Total number of records in the review batch |
| `total_issues` | integer | Yes | Total number of generated issues |
| `issues_by_status` | object | Yes | Counts grouped by `Non-compliant`, `Potentially non-compliant`, and `Review required` |
| `issues_by_risk` | object | Yes | Counts grouped by `High`, `Medium`, and `Low` |
| `issues_by_review_state` | object | Yes | Counts grouped by issue lifecycle state |
| `decisions_by_type` | object | Yes | Counts grouped by decision type |
| `unresolved_issue_count` | integer | Yes | Number of issues not yet closed or accepted |
| `escalated_issue_count` | integer | Yes | Number of escalated issues |
| `summary_note` | string | Yes | High-level summary statement for user-facing output |

## 6. Recommended Optional Fields

The following fields are strongly recommended.

| Field | Type | Required | Description |
|---|---|---|---|
| `source_filename` | string | No | Original file name of the imported dataset |
| `review_period_label` | string | No | Optional human-readable review period label |
| `records_with_issues` | integer | No | Number of unique records with at least one issue |
| `records_without_issues` | integer | No | Number of records with no detected issues |
| `issues_by_category` | object | No | Counts grouped by review category |
| `issues_by_rule` | object | No | Counts grouped by rule ID |
| `corrected_issue_count` | integer | No | Number of issues resolved by correction |
| `false_positive_count` | integer | No | Number of issues closed as false positives |
| `accepted_with_note_count` | integer | No | Number of issues accepted with note |
| `excluded_record_count` | integer | No | Number of records excluded from the review/preparation set |
| `review_completion_rate` | number | No | Percentage of issues no longer in open or in-review state |
| `high_risk_open_count` | integer | No | Number of unresolved high-risk issues |
| `review_started_at` | datetime string | No | Time the review session began |
| `review_completed_at` | datetime string | No | Time the review session was completed, if applicable |
| `is_review_complete` | boolean | No | Whether review is considered complete within the prototype's workflow rules |
| `completion_criteria_note` | string | No | Explanation of how completeness was determined |
| `generated_by` | string | No | Usually `system` |
| `export_format` | string | No | Optional indicator such as `json`, `markdown`, or `ui_view` |

## 7. Canonical Field Definitions

### 7.1 Identification Fields

#### `summary_id`

Unique identifier for the review summary.

Recommended format:

```text
SUM-000001
```

Requirements:

- unique across review summaries
- stable once generated
- not reused

#### `dataset_id`

Identifier of the dataset or review batch that the summary describes.

This should link directly to the imported and normalised dataset used during the review session.

### 7.2 Volume and Coverage Fields

#### `total_records`

Total number of records included in the reviewed dataset.

This should represent the number of normalised records considered by the review process.

#### `records_with_issues`

Number of unique records that produced one or more issues.

This is useful because `total_issues` may be larger than the number of affected records.

#### `records_without_issues`

Number of records with no detected issues under the implemented rule set.

This should not be described as proof of compliance. It only means that no rule in the prototype flagged those records.

#### `total_issues`

Total count of issue objects generated for the dataset.

This should equal the number of issues stored under the issue schema for the review batch.

### 7.3 Classification Summary Fields

#### `issues_by_status`

Counts grouped by issue classification status.

Recommended structure:

```json
{
  "Non-compliant": 5,
  "Potentially non-compliant": 8,
  "Review required": 11
}
```

#### `issues_by_risk`

Counts grouped by issue risk level.

Recommended structure:

```json
{
  "High": 6,
  "Medium": 13,
  "Low": 5
}
```

#### `issues_by_review_state`

Counts grouped by lifecycle state after user interaction.

Recommended structure:

```json
{
  "open": 4,
  "in_review": 2,
  "corrected": 8,
  "accepted_with_note": 5,
  "false_positive": 2,
  "excluded": 1,
  "escalated": 2
}
```

This field is important because it separates detection from review progress.

### 7.4 Category and Rule Summary Fields

#### `issues_by_category`

Counts grouped by review category.

Recommended categories:

- `Required field presence`
- `Date validity and period review`
- `Numeric validity`
- `Duplicate transaction risk`
- `Amount consistency`
- `Reference and evidence traceability`
- `Digital record completeness`
- `Unusual transaction review`

Example:

```json
{
  "Required field presence": 4,
  "Duplicate transaction risk": 3,
  "Unusual transaction review": 2
}
```

#### `issues_by_rule`

Counts grouped by rule ID such as `VR001`, `VR011`, or `VR015`.

This is useful for debugging, evaluation, and dissertation analysis of which rules triggered most often.

### 7.5 Decision Outcome Fields

#### `decisions_by_type`

Counts grouped by recorded user decision.

Recommended structure:

```json
{
  "confirmed_issue": 3,
  "false_positive": 2,
  "corrected": 8,
  "accepted_with_note": 5,
  "excluded_from_review_set": 1,
  "escalated": 2
}
```

This field should be derived from decision log entries where possible rather than entered manually.

#### `corrected_issue_count`

Number of issues resolved by actual data correction.

#### `false_positive_count`

Number of issues closed as false positives.

#### `accepted_with_note_count`

Number of issues accepted after review without correction.

### 7.6 Residual Risk and Completion Fields

#### `unresolved_issue_count`

Number of issues that remain open or insufficiently resolved at the time the summary is generated.

This should usually include issues in:

- `open`
- `in_review`
- optionally `escalated`, depending on how the prototype defines closure

The counting rule should be documented consistently.

#### `high_risk_open_count`

Number of unresolved issues that are also high risk.

This is particularly useful in the prototype because unresolved high-risk items are more important than unresolved low-risk items.

#### `escalated_issue_count`

Number of issues marked as escalated.

Escalated items should be clearly surfaced in the final review summary.

#### `is_review_complete`

Boolean field indicating whether the review process has reached the prototype's defined completion threshold.

Suggested interpretation:

- `true` if there are no open or in-review high-risk issues and the user has finished the current review session
- `false` otherwise

This should not be described as meaning the dataset is fully compliant.

#### `completion_criteria_note`

Optional explanation of how completeness was determined.

Example:

- `Review marked complete because all high-risk issues were closed and remaining medium-risk items were accepted with note.`

### 7.7 Narrative Summary Field

#### `summary_note`

A short user-facing statement summarising the review outcome in plain English.

Examples:

- `24 issues were identified across 18 records. Most high-risk issues were corrected, but 2 escalated items remain for further review.`
- `The dataset has been reviewed under the prototype's implemented checks. Some records remain unresolved and require further manual review before preparation proceeds.`

This field should be descriptive, cautious, and non-legalistic.

## 8. Relationship to Other Schemas

The review summary should be generated from the other core review entities:

- records
- issues
- decision logs

Relationship to project documents:

- `VAT_review_rules.md` defines the checks that generate issues
- `issue_schema.md` defines how issues are stored
- `decision_log_schema.md` defines how user review outcomes are stored
- `VAT_review_workflow.md` defines when the review summary is generated

The review summary should not duplicate full issue or decision details. It should aggregate and summarise them.

## 10. Minimal Schema for an Undergraduate Prototype

If implementation time is limited, the minimum viable review summary schema should contain:

- `summary_id`
- `dataset_id`
- `generated_at`
- `total_records`
- `total_issues`
- `issues_by_status`
- `issues_by_risk`
- `issues_by_review_state`
- `decisions_by_type`
- `unresolved_issue_count`
- `escalated_issue_count`
- `summary_note`

This is enough to support:

- a final review dashboard
- basic exportable summary
- dissertation reporting of review outcomes
- tracking of unresolved items

If possible, also include:

- `records_with_issues`
- `issues_by_category`
- `high_risk_open_count`
- `is_review_complete`

These improve practical usefulness significantly.

## 11. Recommended Validation Rules

The review summary object should also be validated.

Recommended validation constraints:

- `summary_id` must be unique and non-empty
- `dataset_id` must reference an existing dataset
- `generated_at` must be a valid timestamp
- `total_records` must be a non-negative integer
- `total_issues` must be a non-negative integer
- values in `issues_by_status`, `issues_by_risk`, and `issues_by_review_state` must be non-negative integers
- `unresolved_issue_count` must be a non-negative integer
- `escalated_issue_count` must be a non-negative integer
- `summary_note` must not be blank
- if `records_with_issues` and `records_without_issues` are both present, their sum should equal `total_records`
- if `review_completion_rate` is present, it should be between 0 and 100

## 12. Summary Language Guidance

The wording used in the review summary should be careful and bounded.

Recommended wording style:

- `issues were detected`
- `records were reviewed under the prototype's implemented checks`
- `some items remain unresolved`
- `further manual review is required for escalated items`

Avoid wording such as:

- `the dataset is fully compliant`
- `safe to submit`
- `legally correct`
- `VAT treatment confirmed`

The review summary should communicate review progress and residual concerns, not legal certainty.

## 13. Implementation Notes for an Undergraduate Prototype

### 13.1 Generate the summary from stored data

The summary should be derived from:

- the normalised records
- the issue table
- the decision log table

This is better than manually constructing the summary because it preserves consistency.

### 13.2 Keep summary metrics simple and defensible

The most important metrics are:

- total issues
- issues by status
- issues by risk
- unresolved issues
- escalated issues
- corrected issues

These are easy to explain and evaluate.

### 13.3 Surface unresolved high-risk issues clearly

If the prototype only highlights one residual-risk metric prominently, it should be unresolved high-risk issues.

This is more meaningful than a generic percentage alone.

### 13.4 Treat review completion as operational, not legal

If the system supports a field such as `is_review_complete`, it should mean:

- the implemented review workflow has been completed to the system's own criteria

It should not mean:

- the records are definitely compliant
- no further professional judgement is required

## 14. Summary

The review summary schema should provide a structured, high-level account of what happened during pre-submission VAT record review.

A good review summary schema should:

- identify the dataset reviewed
- report how many records and issues were involved
- summarise issue classification and review outcomes
- highlight unresolved and escalated items
- support a cautious narrative summary
- avoid claiming final legal certainty

This makes the schema suitable for a local-first, human-in-the-loop undergraduate prototype where the main goal is structured review support, traceability, and pre-submission transparency.
