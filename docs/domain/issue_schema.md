# Issue Schema for VAT Pre-Submission Review Prototype

## 1. Purpose

This document defines the issue data schema used by the prototype to represent flagged record-level review findings during pre-submission VAT record checking.

The schema is designed to support the following goals:

- store issues generated by deterministic checks and risk-signal checks
- present clear and explainable outputs to the user
- support human-in-the-loop review and correction
- link flagged issues to records, rules, and decision logs
- preserve traceability from detection through review outcome

This schema supports a pre-submission VAT record review workflow. It is not intended to represent final tax judgement, final VAT liability, or legal compliance certification.

## 2. Design Principles

The issue schema should follow these principles:

- **Explainability**: each issue must state what was detected, why it matters, and what should be checked manually.
- **Traceability**: each issue must be linkable to a source record, rule, timestamp, and later review decision.
- **Separation of machine output and human judgement**: the issue stores the system's detection result, while the user's review outcome is recorded separately.
- **Implementation simplicity**: the schema should be practical for a local-first undergraduate prototype using CSV/Excel inputs and JSON/SQLite/file-based persistence.
- **Bounded scope**: the issue schema should support review and correction workflow, not full accounting or automated tax decision-making.

## 3. Schema Role in the Workflow

Within the system workflow, an issue object is created after a rule detects a problem or risk signal.

The issue object should support:

1. linking the issue to the relevant record
2. describing the detected problem
3. classifying the issue by category, status, and risk
4. providing user-facing explanation
5. suggesting a manual check
6. tracking review progress
7. linking to user decision logging
8. contributing to a pre-submission review summary

## 4. Core Entity Definition

An issue represents one detected problem, warning, or review signal for one record under one rule.

A new issue should normally be created when:

- one rule triggers on one record
- the trigger is significant enough to present to the user
- the issue requires visibility, review, correction, or logging

A single record may have multiple issues.

For example, one row may simultaneously have:

- invalid date format
- missing supplier reference
- unusual net amount

These should normally be stored as separate issues because they differ in logic, explanation, and review action.

## 5. Required Fields

Each issue should include the following required fields.

| Field | Type | Required | Description |
|---|---|---|---|
| `issue_id` | string | Yes | Unique identifier for the issue instance |
| `rule_id` | string | Yes | Identifier of the rule that triggered the issue |
| `record_id` | string | Yes | Identifier of the affected record |
| `issue_type` | string | Yes | Machine-readable issue label |
| `category` | string | Yes | Review category for grouping and explanation |
| `status` | enum | Yes | Review status classification |
| `risk_level` | enum | Yes | Risk level classification |
| `determinism_type` | enum | Yes | Whether detection is deterministic, partly deterministic, or manual-only |
| `detection_summary` | string | Yes | Short explanation of what the system detected |
| `why_it_matters` | string | Yes | Why the issue matters in pre-submission VAT review |
| `possible_vat_review_impact` | string | Yes | Possible consequence for VAT review/preparation |
| `recommended_manual_check` | string | Yes | Action the user should take to review or resolve the issue |
| `review_state` | enum | Yes | Workflow state of the issue |
| `detected_at` | datetime string | Yes | Timestamp when the issue was generated |

## 6. Recommended Optional Fields

The following fields are strongly recommended because they improve usability, explainability, and traceability.

| Field | Type | Required | Description |
|---|---|---|---|
| `dataset_id` | string | No | Identifier of the imported dataset or review batch |
| `row_index` | integer | No | Original row number from the imported spreadsheet |
| `field_names` | array of strings | No | Names of fields involved in the issue |
| `detected_value` | object, string, number, or null | No | Value or field snapshot relevant to detection |
| `expected_value` | string, number, object, or null | No | Optional expected or valid form |
| `severity_score` | integer or number | No | Optional numeric score for sorting or prioritisation |
| `review_priority` | integer | No | Optional queue priority |
| `evidence_expected` | string | No | Type of evidence likely to help resolve the issue |
| `source_snapshot` | object | No | Small snapshot of relevant record fields at detection time |
| `period_context` | object | No | Optional period metadata relevant to review |
| `is_resolved` | boolean | No | Convenience field derived from `review_state` |
| `resolved_at` | datetime string | No | Timestamp when the issue was resolved |
| `resolution_reason` | string | No | Optional short summary of how the issue was resolved |
| `linked_decision_id` | string | No | ID of associated review decision log entry |
| `created_by` | string | No | Typically `system` |
| `updated_at` | datetime string | No | Last update time for the issue object |

## 7. Enumerations

### 7.1 `status`

Allowed values:

- `Non-compliant`
- `Potentially non-compliant`
- `Review required`

Definitions:

- `Non-compliant`: the record has a clear and directly detectable issue that prevents it from being treated as review-ready in its current form.
- `Potentially non-compliant`: the system detected a meaningful risk signal, but cannot conclude from spreadsheet data alone that the record is definitely wrong.
- `Review required`: the record is not clearly invalid, but additional user review is needed before confidence is sufficient.

### 7.2 `risk_level`

Allowed values:

- `High`
- `Medium`
- `Low`

Definitions:

- `High`: likely to materially affect review reliability, completeness, or preparation quality
- `Medium`: introduces meaningful uncertainty or traceability weakness
- `Low`: limited direct impact, but still useful to flag for completeness or transparency

### 7.3 `determinism_type`

Allowed values:

- `Deterministic`
- `Partly deterministic`
- `Manual only`

Definitions:

- `Deterministic`: the issue was detected using a concrete rule with no major interpretive step
- `Partly deterministic`: the issue was detected by a concrete rule, but the result should be treated as a risk signal rather than a definitive failure
- `Manual only`: the issue cannot be concluded by system logic alone and exists primarily as a prompt for structured review

### 7.4 `review_state`

Allowed values:

- `open`
- `in_review`
- `resolved`
- `accepted_with_note`
- `corrected`
- `false_positive`
- `excluded`
- `escalated`

Recommended use:

- `open`: issue has been generated but not yet reviewed
- `in_review`: user has started assessing it
- `resolved`: general resolved state if a more specific state is not needed
- `accepted_with_note`: user determined the record is acceptable after review
- `corrected`: issue was resolved by changing the record
- `false_positive`: user determined the issue should not have been treated as a problem
- `excluded`: the record was excluded from the review/preparation set
- `escalated`: issue requires review outside the prototype

## 8. Canonical Field Definitions

### 8.1 Identification Fields

#### `issue_id`

A unique identifier for the issue instance.

Recommended format:

```text
ISSUE-000001
```

Requirements:

- unique within the dataset or storage system
- stable once assigned
- not reused

#### `rule_id`

Identifier of the rule that generated the issue.

Examples:

- `VR001`
- `VR011`
- `VR015`

This field is important for:

- explaining the logic behind the issue
- linking issues to rule metadata
- reporting issue counts by rule

#### `record_id`

Identifier of the record affected by the issue.

This should link directly to the normalised record entity used internally by the prototype.

Recommended format:

```text
ROW-45
```

or

```text
REC-2026-00045
```

### 8.2 Classification Fields

#### `issue_type`

A short machine-readable label for the issue.

Examples:

- `missing_transaction_date`
- `invalid_date_format`
- `missing_net_amount`
- `non_numeric_vat_amount`
- `duplicate_invoice_reference`
- `exact_duplicate_row`
- `inconsistent_totals`
- `missing_evidence_reference`
- `unusual_net_amount`

This field is useful for filtering, UI grouping, and downstream summary generation.

#### `category`

A higher-level grouping used in the UI and summary reports.

Recommended values:

- `Required field presence`
- `Date validity and period review`
- `Numeric validity`
- `Duplicate transaction risk`
- `Amount consistency`
- `Reference and evidence traceability`
- `Digital record completeness`
- `Unusual transaction review`

### 8.3 Explanation Fields

#### `detection_summary`

A short plain-English explanation of what the system found.

Examples:

- `Transaction date is missing.`
- `VAT amount contains a non-numeric value.`
- `Gross amount does not reconcile with net amount plus VAT amount.`
- `Invoice reference appears more than once in the dataset.`

This should describe detection, not interpretation.

#### `why_it_matters`

Explains why the issue is relevant in the pre-submission review workflow.

Examples:

- `A missing transaction date makes it harder to review whether the record belongs in the correct period.`
- `A non-numeric VAT amount reduces confidence in the record and may affect preparation accuracy.`

This should connect the issue to review quality and record integrity.

#### `possible_vat_review_impact`

Explains the likely practical impact on review or preparation if the issue is left unresolved.

Examples:

- `The transaction may be reviewed in the wrong period or omitted from correct checking.`
- `The record may need correction before it can be treated as review-ready.`

This should remain cautious and should not claim definitive legal consequences unless the system can justify them.

#### `recommended_manual_check`

A specific next action for the user.

Examples:

- `Check the original invoice or ledger export and enter the correct transaction date.`
- `Compare the duplicated records and confirm whether one should be removed or annotated.`
- `Review the supporting evidence and confirm whether the unusual amount is valid.`

This field is important because the prototype is explicitly human-in-the-loop.

## 9. Context Fields

### 9.1 `field_names`

List the fields involved in the issue.

Examples:

```json
["transaction_date"]
```

```json
["net_amount", "vat_amount", "gross_amount"]
```

This helps the UI highlight affected columns and supports explainability.

### 9.2 `detected_value`

Stores the problematic value or values as detected by the system.

Examples:

```json
"31/02/2026"
```

```json
{
  "net_amount": 100.00,
  "vat_amount": 20.00,
  "gross_amount": 150.00
}
```

This field should capture enough context to understand the issue without requiring immediate re-reading of the entire row.

### 9.3 `expected_value`

Optional field describing what would have been valid or expected.

Examples:

```json
"A valid parseable date"
```

```json
{
  "gross_amount": "net_amount + vat_amount"
}
```

This can help the user understand how to resolve the issue.

### 9.4 `source_snapshot`

An optional lightweight snapshot of the relevant row at issue creation time.

Example:

```json
{
  "transaction_date": "",
  "invoice_reference": "INV-2045",
  "description": "Office equipment",
  "net_amount": 350.00,
  "vat_amount": 70.00
}
```

This is useful if the underlying row later changes and the system still needs an auditable record of what was originally flagged.

## 10. Workflow and Resolution Fields

### 10.1 `review_state`

Tracks where the issue sits in the user review process.

Recommended state flow:

```text
open -> in_review -> corrected
open -> in_review -> accepted_with_note
open -> in_review -> false_positive
open -> in_review -> escalated
open -> in_review -> excluded
```

A simplified implementation may also support:

```text
open -> resolved
```

### 10.2 `is_resolved`

Convenience field indicating whether the issue is no longer pending.

Recommended mapping:

- `true` for `resolved`, `corrected`, `accepted_with_note`, `false_positive`, `excluded`
- `false` for `open`, `in_review`, `escalated`

### 10.3 `resolved_at`

Timestamp when the issue reached a closed or terminal review state.

### 10.4 `resolution_reason`

Short human-readable summary of the outcome.

Examples:

- `Date corrected from invoice copy`
- `Duplicate reference confirmed as split-line posting`
- `Outlier accepted after evidence review`

### 10.5 `linked_decision_id`

Reference to the associated review decision log entry.

This should support one of two practical models:

- one issue linked to one decision entry
- one issue linked to the latest decision entry, with full history stored elsewhere

For the prototype, a single latest-decision link is sufficient.

## 12. Minimal Schema for an Undergraduate Prototype

If implementation time is limited, the minimum viable issue schema should contain:

- `issue_id`
- `rule_id`
- `record_id`
- `issue_type`
- `category`
- `status`
- `risk_level`
- `determinism_type`
- `field_names`
- `detection_summary`
- `why_it_matters`
- `possible_vat_review_impact`
- `recommended_manual_check`
- `review_state`
- `detected_at`

This minimal set is enough to support:

- rule output display
- issue filtering
- review workflow
- summary generation
- basic traceability

## 13. Recommended Validation Rules for the Issue Object

The issue object itself should also be validated.

Recommended validation constraints:

- `issue_id` must be non-empty and unique
- `rule_id` must match a known rule definition
- `record_id` must match an existing record
- `status` must be one of the allowed enum values
- `risk_level` must be one of the allowed enum values
- `determinism_type` must be one of the allowed enum values
- `review_state` must be one of the allowed enum values
- required explanation fields must not be blank
- `detected_at` must be a valid timestamp
- if `is_resolved = true`, then `review_state` should be a terminal state
- if `resolved_at` is present, it should not be earlier than `detected_at`

## 14. Relationship to Other Entities

The issue schema should be treated as one part of a larger review data model.

Recommended entity relationships:

- one dataset has many records
- one record can have many issues
- one rule can generate many issues
- one issue can have one current linked decision record
- one issue may contribute to one review summary

Suggested conceptual links:

```text
Dataset -> Records -> Issues -> Decision Logs -> Review Summary
```

The issue object should store the machine-detected finding. The user decision log should store the human review outcome.

## 15. Field Naming Guidance

For implementation clarity, field names should be:

- lowercase
- snake_case
- stable across JSON export, internal storage, and UI mapping

Recommended style:

- `issue_type`
- `risk_level`
- `recommended_manual_check`

Avoid mixing styles such as camelCase and snake_case in the same prototype.

## 16. Implementation Notes

### 16.1 Keep issue generation rule-driven

Each issue should be generated from a known rule definition rather than ad hoc logic embedded directly in the UI.

This makes the system easier to:

- test
- explain
- maintain
- summarise by rule/category

### 16.2 Keep issue text templated

For consistency, explanation fields should preferably be generated from rule templates with record-specific values inserted where useful.

Example pattern:

- detection summary template
- why it matters template
- possible impact template
- recommended manual check template

This improves consistency and reduces arbitrary wording.

### 16.3 Preserve original detection context

If possible, the system should keep enough context to show what was flagged at the time of detection, even if the record is later edited.

This is useful for traceability and dissertation discussion of review transparency.

### 16.4 Do not treat issue objects as final truth

An issue object represents a flagged concern, not a final legal or tax conclusion.

This distinction should be reflected in:

- field naming
- UI wording
- review workflow
- summary language

## 17. Suggested Output for User Interface

At the UI level, an issue card or issue table should at minimum display:

- issue type
- record identifier
- status
- risk level
- detection summary
- why it matters
- recommended manual check
- current review state

Optional helpful UI fields:

- affected fields
- detected values
- linked evidence expectation
- timestamp
- linked decision status

This supports the prototype's core aim of structured, explainable, pre-submission review.

## 18. Summary

The issue schema should be designed as a structured representation of detected review findings, not as a final tax judgement object.

A good prototype issue schema should:

- link each issue to a rule and a record
- classify the issue by status and risk
- explain the issue clearly
- suggest a concrete manual check
- support review-state tracking
- link to decision logging
- preserve traceability from detection to outcome

This makes the schema suitable for a local-first, human-in-the-loop VAT pre-submission review prototype focused on explainability, traceability, and practical implementation.
